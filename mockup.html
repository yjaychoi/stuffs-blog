Paths of MySQL, vector search edition

In the ever-evolving landscape of database technology, the integration of vector search capabilities has become a pivotal feature for modern applications. As we move deeper into 2026, the demand for semantic search powered by large language models continues to reshape how we architect data storage layers.

MySQL, often seen as the traditional workhorse of the web, hasn't been sitting idle. The ecosystem has bifurcated into several interesting paths, each attempting to solve the k-nearest neighbor problem without sacrificing the ACID compliance we all know and love.

The Native Approach

Initially, many developers resorted to storing vectors as BLOB types and performing brute-force scans. While simple, this approach scales linearly with data size—a death knell for latency-sensitive applications.

sql_example.sql 
CREATE TABLE embeddings (
    id INT PRIMARY KEY AUTO_INCREMENT,
    content TEXT,
    embedding VECTOR(1536) -- New MySQL 9.0 syntax
);
-- Performing a cosine similarity search
SELECT id, content,
       vec_cosine_distance(embedding, @query_vector) AS distance
FROM embeddings
ORDER BY distance ASC
LIMIT 5;

The snippet above highlights the new native syntax introduced recently. It’s clean, integrated, and surprisingly performant for datasets under a few million rows. However, when you start hitting the 10M+ row mark, the lack of an HNSW index becomes apparent.

Alternative Extensions

This is where extensions like pgvector's spiritual successors in the MySQL world come into play. We are seeing a renaissance of storage engines designed specifically for high-dimensional data.

  • PlanetScale's Vector: A serverless-first approach that abstracts the indexing complexity.
  • Manticore Search: While not strictly MySQL, it speaks the protocol and offers lightning-fast retrieval.
